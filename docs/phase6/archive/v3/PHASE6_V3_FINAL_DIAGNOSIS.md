## 根本问题分析

### 可能的原因

#### 1. 动力学模型不匹配（最可能）
**证据**：
- MPC 优化出的力矩在 MuJoCo 中执行后，实际状态轨迹与 MPC 预测的差异很大
- 即使 MPC "优化"到了某个解，这个解在实际系统上也不适用

**原因**：
- ALIGATOR 使用 Pinocchio 的动力学（从 MJCF 加载）
- MuJoCo 有自己的动力学实现
- 两者可能在以下方面不同：
  - 惯性参数精度
  - 摩擦模型
  - 数值积分方法
  - 接触模型

#### 2. 问题规模太大
- 状态维度：32 (16 q + 16 v)
- 控制维度：16
- Horizon：10
- 总决策变量：10 × 16 = 160
- 对于 ProxDDP，这不算特别大，但双臂系统确实复杂

#### 3. 末端执行器代价的 Jacobian 问题
- `FrameTranslationResidual` 需要计算 Jacobian
- 对于双臂系统，Jacobian 可能病态
- 导致优化方向不准确

#### 4. 初始化仍然太差
- 即使有 warm start，如果上一次就没收敛，warm start 也是坏的
- 陷入局部最小值

## 对比：为什么纯 PD 性能好？

**纯 PD（1.79cm）的优势：**
1. **模型无关**：不依赖动力学模型
2. **简单有效**：直接根据位置/速度误差计算力矩
3. **鲁棒性强**：对模型误差不敏感
4. **计算快**：无需优化

**动力学 MPC 的劣势：**
1. **依赖模型准确性**：模型不准→优化无意义
2. **计算复杂**：需要收敛才有效
3. **容易失败**：不收敛时性能急剧下降

## 建议的替代方案

### 方案 A：运动学 MPC + 逆动力学
```
运动学 MPC（优化关节轨迹 q_des）
    ↓
逆动力学（计算所需力矩 τ）
    ↓
PD 控制（跟踪 q_des，前馈 τ）
```
**优点**：
- 运动学 MPC 更容易收敛（无动力学约束）
- 逆动力学快速准确
- 仍然是力矩控制

### 方案 B：保持纯 PD
```
IK 规划器（生成 q_des 轨迹）
    ↓
PD 控制（跟踪 q_des）
```
**优点**：
- 已验证性能优秀（1.79cm）
- 简单可靠
- 实时性好

**缺点**：
- 无法优化动力学
- 无法考虑力矩限制（但实际中 PD 本身已处理）

### 方案 C：模型学习 + MPC
```
数据驱动模型学习（修正 Pinocchio 模型）
    ↓
动力学 MPC（使用修正后的模型）
```
**优点**：
- 解决模型不匹配问题
- 保留 MPC 的优化能力

**缺点**：
- 需要大量数据
- 工程复杂度高

## 结论

1. **所有提出的结构性修复都已完成**，代码质量已显著提升
2. **MPC 完全不收敛（0%）** 是核心问题，不是调参能解决的
3. **最可能的原因是动力学模型不匹配**
4. **纯 PD 控制已经很好（1.79cm）**，不需要强行使用 MPC
5. **建议：**
   - **短期**：使用方案 B（纯 PD），性能已验证
   - **中期**：如果需要优化，尝试方案 A（运动学 MPC）
   - **长期**：如果必须用动力学 MPC，需要方案 C（模型学习）
