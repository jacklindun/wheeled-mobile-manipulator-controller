# 任务规划文档：基于 MuJoCo + ALIGATOR 的轮式 UR5e 机械臂运动学 MPC 全身控制器

## 1. 项目目标

本项目需要从零构建一个简单的轮式机械臂仿真系统，并在 MuJoCo 中实现基于 ALIGATOR 的 MPC 全身控制器。

第一版目标不是做完整力矩级全动力学 MPC，而是先跑通一个完整、稳定、可视化的 baseline：

1. 使用 MuJoCo 原生 MJCF XML 搭建轮式机械臂模型；
2. 机械臂直接使用现成 UR5e MuJoCo MJCF 模型；
3. 不创建 URDF，不做 URDF 到 MJCF 的转换；
4. 底盘使用虚拟全向移动底盘；
5. 底盘加入 `base_z` 升降自由度；
6. MPC 同时控制：

   * 底盘 `base_x`
   * 底盘 `base_y`
   * 底盘 `base_z`
   * 底盘 `base_yaw`
   * UR5e 六个关节
7. MPC 解算器必须使用 ALIGATOR；
8. 第一版 MPC 为 kinematic whole-body MPC；
9. MPC 输出底盘速度和 UR5e 关节速度；
10. MuJoCo 低层使用 position servo 或 velocity servo 执行；
11. 在 MuJoCo viewer 中展示末端轨迹跟踪效果；
12. 记录 MPC 求解时间、末端误差、控制输入，并生成结果图。

第一版系统可以理解为：

```text
MuJoCo MJCF 模型
+
UR5e 现成机械臂
+
4DoF 虚拟移动升降底盘
+
ALIGATOR 运动学 MPC
+
MuJoCo 闭环仿真可视化
```



## 5. 运动学模型

ALIGATOR MPC 使用离散运动学模型。

状态：

```text
x = q
```

控制：

```text
u = qdot_command
```

离散模型：

```text
base_x_next =
  base_x + dt * (cos(base_yaw) * base_vx_body - sin(base_yaw) * base_vy_body)

base_y_next =
  base_y + dt * (sin(base_yaw) * base_vx_body + cos(base_yaw) * base_vy_body)

base_z_next =
  base_z + dt * base_vz

base_yaw_next =
  base_yaw + dt * base_omega

q_arm_next =
  q_arm + dt * qd_arm
```

注意：

1. `base_vx_body` 和 `base_vy_body` 是底盘自身坐标系下速度；
2. `base_x` 和 `base_y` 是世界坐标系位置；
3. 需要通过 yaw 将 body-frame velocity 转换成 world-frame velocity；
4. `base_z` 是竖直升降自由度；
5. UR5e 六个关节直接一阶积分；
6. `base_yaw` 需要 wrap 到 `[-pi, pi]`。

## 10. ALIGATOR MPC 总体设计

第一版 MPC 是有限时域最优控制问题：

```text
minimize over x_0...x_N, u_0...u_{N-1}

Σ l_k(x_k, u_k, ref_k) + l_N(x_N, ref_N)

subject to:
x_0 = x_measured
x_{k+1} = f(x_k, u_k)
x_min <= x_k <= x_max
u_min <= u_k <= u_max
```

其中：

```text
x = q
u = qdot_command
```

需要使用 ALIGATOR 构建和求解这个问题。

### 11.2 dynamics model

需要创建一个 ALIGATOR 可使用的 discrete dynamics model，对应：

```text
x_{k+1} = f(x_k, u_k)
```

第一版可以使用以下方式之一：

#### 方案 A：自定义 Python dynamics model

如果 ALIGATOR Python API 支持自定义 dynamics/cost model，则实现：

```python
class WheeledUR5eKinematicDynamics:
    def __init__(self, robot_model, dt):
        ...

    def forward(self, x, u):
        return robot_model.dynamics_numpy(x, u, dt)

    def derivatives(self, x, u):
        A, B = robot_model.linearize_dynamics(x, u, dt)
        return A, B
```

#### 方案 B：使用 ALIGATOR 内置 explicit dynamics / integrator 接口

如果 ALIGATOR 有内置 explicit dynamics 或 discrete dynamics wrapper，请使用官方接口封装 `f(x,u)`。

#### 方案 C：暂时构建线性时变近似

如果自定义非线性 dynamics 接口实现难度较大，可以先在每个 MPC step 中围绕当前 warm-start trajectory 构建局部线性模型：

```text
x_{k+1} = A_k x_k + B_k u_k + c_k
```

但要在 README 中说明这是第一版近似，后续替换为完整非线性 dynamics model。

优先级：

```text
方案 A > 方案 B > 方案 C
```

### 11.3 cost model

每个 stage 的 cost：

```text
l_k =
  w_ee_pos   * || p_ee(x_k) - p_ee_ref_k ||^2
+ w_base_xy  * || base_xy_k - base_xy_ref_k ||^2
+ w_base_yaw * angle_error(base_yaw_k, base_yaw_ref_k)^2
+ w_base_z   * || base_z_k - base_z_ref_k ||^2
+ w_posture  * || q_arm_k - q_arm_nominal ||^2
+ w_u        * || u_k ||^2
+ w_du       * || u_k - u_{k-1} ||^2
```

terminal cost：

```text
l_N =
  w_terminal_ee * || p_ee(x_N) - p_ee_ref_N ||^2
+ w_terminal_posture * || q_arm_N - q_arm_nominal ||^2
```

推荐权重：

```python
weights = {
    "ee_pos": 100.0,
    "terminal_ee": 200.0,
    "base_xy": 3.0,
    "base_yaw": 1.0,
    "base_z": 5.0,
    "posture": 0.5,
    "terminal_posture": 0.5,
    "u": 0.01,
    "du": 0.1,
}
```

如果 ALIGATOR 支持 residual/cost stacking，请把上述 cost 拆成多个 residual，方便调试。

### 11.4 constraints

必须支持：

```text
x_min <= x_k <= x_max
u_min <= u_k <= u_max
```

即：

```text
q_min <= q <= q_max
u_min <= u <= u_max
```

如果 ALIGATOR 当前 Python API 中加入 box constraints 不方便，则第一版可以先用 soft constraint penalty 实现，但代码中必须留出 hard constraint 接口，并在 README 里标注：

```text
当前版本将 q/u limits 作为 soft penalty，下一版改为 ALIGATOR hard constraints。
```

推荐尽量实现 hard bounds。

### 11.5 reference update

MPC 每一帧参考轨迹都会变化。

需要保证：

1. 不要每一帧重复构建大量不必要对象；
2. 如果 ALIGATOR 支持 parameter update，应优先更新 reference 参数；
3. 如果第一版 API 不熟悉，可以每一帧重建 problem，但需要记录 solve time；
4. 后续优化为固定 problem structure + 更新 reference。

或当前版本中等价求解器。

推荐 solver 配置：

```text

verbose = False
```

实际参数按当前 ALIGATOR API 调整。

### 12.3 real-time iteration 模式

第一版为了闭环实时性，不需要每一帧完全收敛，可以采用类似 real-time iteration 的方式：

```text
每个 MPC cycle 做固定少量迭代
例如 5 到 10 次
```

需要在参数中支持：

```python
self.max_iters_per_mpc = 10
```

如果求解时间太慢，可以降到 3 到 5。

### 12.4 warm start

必须实现 warm start：

1. 保存上一帧的 state trajectory；
2. 保存上一帧的 control trajectory；
3. 下一帧将轨迹整体 shift 一步；
4. 最后一帧用最后一个状态/控制补齐；
5. 用 shift 后的轨迹作为 ALIGATOR 初值。

伪代码：

```python
def shift_warm_start(xs_prev, us_prev):
    xs_init = xs_prev[1:] + [xs_prev[-1]]
    us_init = us_prev[1:] + [us_prev[-1]]
    return xs_init, us_init
```

没有上一帧时：

```text
xs_init 全部使用 q_current
us_init 全部使用 0 或 u_prev
```

```python
def solve(self, q_current, ref_traj, u_prev=None):
    """
    输入:
```

### 12.6 求解失败 fallback

不能因为 ALIGATOR 求解失败导致仿真崩溃。

如果失败：

1. 使用上一帧成功的 `u0`；
2. 如果上一帧也没有，使用零控制；
3. 标记 `fallback=True`；
4. 继续仿真；
5. logger 记录失败状态。

## 21. 常见问题和处理策略

### 21.1 ALIGATOR import 失败

处理：

1. 检查 Python 环境；
2. 检查是否安装 `aligator`；
3. 检查是否需要 conda-forge；
4. 检查是否和 Pinocchio/eigenpy 版本冲突；
5. README 中写清楚安装方式。

### 21.2 ALIGATOR API 不匹配

处理：

1. 优先查看本地安装的 ALIGATOR examples；
2. 打印 `aligator.__version__`；
3. 不要硬套网上旧 API；
4. 在代码注释中记录当前 API；
5. 先实现最小 OCP，再扩展到完整 MPC。

### 21.3 ALIGATOR 求解太慢

处理：

1. horizon 从 20 降到 10；
2. max_iters 从 10 降到 5；
3. 降低轨迹速度；
4. 减少 cost 项；
5. 使用 warm start；
6. 避免每帧重复构建复杂对象；
7. 尽量固定 problem structure，只更新 reference。

### 21.4 ALIGATOR 不收敛

处理：

1. 降低参考轨迹速度；
2. 增大 `w_u` 和 `w_du`；
3. 减小 `w_ee_pos`；
4. 检查 FK 是否和 MuJoCo 一致；
5. 检查 q/u bounds 是否过紧；
6. 检查初始状态是否可行；
7. 使用上一帧轨迹 warm start。

### 21.5 FK 和 MuJoCo 末端位置不一致

这是关键问题。必须检查：

1. UR5e 连杆参数是否和 MJCF 一致；
2. 关节轴方向是否一致；
3. 关节顺序是否一致；
4. base yaw 和 base translation 是否正确进入 FK；
5. base_z 是否正确叠加到末端世界坐标；
6. ee_site 的位置是否和 FK 末端定义一致。

如果 FK 和 MuJoCo 不一致，MPC 会优化一个错误模型，闭环效果会很差。

### 21.6 控制抖动

处理：

1. 增大 `w_du`；
2. 降低速度上限；
3. 降低轨迹速度；
4. 增大 MuJoCo position actuator damping；
5. 检查 `q_des` 是否跳变；
6. 检查 yaw angle wrap。

## 23. 当前版本局限性

请在 README 中明确说明：

1. 当前是 kinematic MPC，不是 full dynamics MPC；
2. 当前使用 ALIGATOR 解算运动学 OCP；
3. 轮子只是视觉模型，不是真实滚动接触；
4. 底盘通过虚拟 slide/hinge joint 移动；
5. 控制器输出速度，不输出力矩；
6. 低层使用 position servo，不是 torque control；
7. 没有碰撞约束；
8. 没有末端姿态控制，只控制末端位置；
9. 没有真实轮地动力学；
10. 没有 inverse dynamics QP；
11. FK 必须和 MuJoCo MJCF 保持一致，否则控制效果会受影响。

## 24. 下一步升级建议

### V2：末端姿态跟踪

当前只跟踪末端位置，后续加入末端姿态：

```text
SO(3) orientation error
```

MPC cost 加入：

```text
w_ee_ori * orientation_error(R_ee, R_ref)^2
```

### V3：kinodynamic MPC

状态扩展为：

```text
x = [q, qdot]
```

控制量变为：

```text
u = qddot
```

这样可以自然惩罚速度、加速度，使轨迹更平滑。

### V4：真实全向轮运动学

加入四个轮子的实际转速：

```text
wheel_speed = A @ [vx, vy, omega]
```

MuJoCo 中轮子真正旋转，而不是只作为视觉模型。

### V5：ALIGATOR full dynamics MPC

从运动学模型升级到动力学模型：

```text
x = [q, v]
u = tau
```

使用 Pinocchio / ALIGATOR 构建完整动力学模型。

### V6：WBC / inverse dynamics QP

MPC 输出期望：

```text
q_ref, qdot_ref, qddot_ref
```

低层使用：

```text
inverse dynamics QP
或 operational-space control
```

计算力矩。

### V7：加入 body 更多自由度

从当前：

```text
base_x, base_y, base_z, base_yaw
```

扩展为：

```text
base_x, base_y, base_z, base_roll, base_pitch, base_yaw
```

或者加入类似机器人腰部结构的自由度。

### V8：双臂扩展

从单 UR5e 扩展到：

```text
mobile base + body + left_arm + right_arm
```

为后续轮式双臂机器人全身控制做准备。
